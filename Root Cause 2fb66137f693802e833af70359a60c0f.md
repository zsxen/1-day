# Root Cause

Date: March 24, 2026 5:27 PM
Status: In progress

## Current confirmed / unconfirmed evidence

Confirmed so far:

- `SepDuplicateToken` is the main dynamic-analysis target, and the current notes identify a patch-shift where EffectiveOnly handling appears to move from a separate `SepMakeTokenEffectiveOnly` call into the caller-side flow before later SID hash rebuild work.
- The current notes identify `ExReleaseResourceLite` as the key synchronization boundary to validate, especially the candidate `mov rcx, [rdi+30h] ; Resource` pattern near the suspicious pre-patch flow.
- `SepMakeTokenEffectiveOnly` has been reached in kernel debugging with a basic breakpoint that logs `rcx` as a token candidate and the current thread.

Unconfirmed / needs debugger evidence:

- Whether `rcx` at `SepDuplicateToken` entry is definitely OldToken on Windows 11 26100.7019.
- Whether the token passed to `SepMakeTokenEffectiveOnly` is OldToken, NewToken, or another derived token.
- Whether the resource released at the `SepDuplicateToken` `ExReleaseResourceLite` call-site equals the SMTEO token's resource field.
- Whether `UserAndGroups`, SID entry pointers, count, and index-like fields are observed in an inconsistent intermediate state across concurrent threads.
- Whether any crash-only double-free/UAF evidence can be tied to the same token, `UserAndGroups` array, or inner SID pointer.

# SepDuplicateToken

OldToken을 읽기 lock으로 보호한 채, 새 토큰 오브젝트를 커널 오브젝트로 생성하고, 고정 필드를 복사/설정한다. 그 후, 가변 데이터를 정책에 맞춰 복제하고, 마지막에 SID 해시를 다시 만들고 반환하는 함수이다.

<aside>

1. 입력 파라미터 검사 (특정 조합은 즉시 실패)
2. 새 토큰에 붙을 부가 구조체(권한/추적/리소스 등) 메모리 확보
3. **토큰을 “커널 오브젝트”로 생성**(Obp* 계열 호출)
4. 새 토큰의 핵심 필드(식별자/타입/참조대상/플래그/세션 등) 초기화
5. OldToken에서 가변 길이 데이터(권한, 그룹/사용자 SID 배열, 제한 SID, capability, claim, appcontainer 등) 복제
6. SID hash 등 캐시 재구성
7. 옵션에 따라 “EffectiveOnly” 등 후처리
8. 성공 시 `a8 = NewToken` 설정 후 `STATUS_SUCCESS`
</aside>

## ExReleaseResourceLite

다음은 `ExReleaseResourceLite`이다. 여기도 보면 스핀락이 언급되어 있다. 

https://learn.microsoft.com/en-us/windows-hardware/drivers/ddi/wdm/nf-wdm-exreleaseresourcelite

![image.png](Root%20Cause/image%205.png)

> 스핀락은 대충 lock이 잠겨 있으면 그게 풀릴 때까지 무한 루프를 돌면서 검사한다. 타임 슬라이스가 소진되면 스레드는 컨텍스트 스위칭이 일어나고, 다시 또 스레드가 스케줄되면 락이 풀릴 때까지 검사를 반복한다. 만약에 잠긴 스레드에 대해 여러 스레드가 락을 검사하고 있다면, 락이 해제된 시점 이후로 가장 먼저 락을 검사한 스레드가 락을 잠그고 임계구역에 진입한다.
> 

얘도 내부 코드를 ㅈㅍㅌ에게 맡겨보면,

`ExReleaseResourceLite`는 ERESOURCE(Executive Resource)를 해제하는 함수라고 한다. 그리고 내가 언급했던 내부에 보이는 스핀락은 ****ERESOURCE 구현을 위한 내부 락으로, 지금 멀티스레드 체제에서 보호해야 하는 대상에 대한 락은 아니다. 

그니까 임계구역? 데이터?에 대한 락이 풀린다는 소리…

걸어둔 문서에도 보면 **ExReleaseResourceLite routine releases a specified executive resource owned by the current thread**라고 되어 있다.

그래서 얘를 호출하게 되면 보호가 풀리니까 다른 스레드들도 동일 자원에 접근할 수 있다는 말이 된다. 문서에 리마크 부분도 보면..

<aside>

Normal kernel APC delivery must be disabled before calling this routine. Disable normal kernel APC delivery by calling [KeEnterCriticalRegion](https://learn.microsoft.com/en-us/windows-hardware/drivers/ddi/ntddk/nf-ntddk-keentercriticalregion). Delivery must remain disabled until the resource is released, at which point it can be reenabled by calling [KeLeaveCriticalRegion](https://learn.microsoft.com/en-us/windows-hardware/drivers/ddi/ntddk/nf-ntddk-keleavecriticalregion). For more information, see [Disabling APCs](https://learn.microsoft.com/en-us/windows-hardware/drivers/kernel/disabling-apcs).

</aside>

일반적으로 `KeEnterCriticalRegion()`~`KeLeaveCriticalRegion()`로 APC를 막고 쓰라는, 임계구역에 대한 설명도 붙어 있다. 그니까 작업의 끝까지 안 막아뒀기 때문 아닐까?

이 함수에 딸린 나머지 페어도 봐 보자.. 

### ExInitializeResourceLite

ERESOURCE 구조체를 초기화해서 동기화에 쓰이는 객체(락 변수)를 준비하는 단계이다. 그니까 새 토큰이 앞으로 사용할 락을 준비(생성이랑 초기화)만 하는 것이라 임계구역이랑은 관련이 없다.

### ExAcquireResourceSharedLite

얘가 하나 더 있었는데 몰랐다. 근데 얘가 진짜였다.

https://learn.microsoft.com/ko-kr/windows-hardware/drivers/ddi/wdm/nf-wdm-exacquireresourcesharedlite

ERESOURCE를 공유 모드로 획득하는 단계로, 현재 스레드가 해당 리소스에 대해 공유 락을 걸고 임계구역에 들어간다. 그리고 보호가 필요한 코드 구간마다 이 함수를 반복적으로 호출한다. 그리고 반드시 나중에 `ExReleaseResourceLite`로 해제해야 한다.

그러니까 이제…얘네가 호출되는 부분을 따라서 분기를 타보자. 해제 함수 전후로 뭔가 의심점이 있는지 각 lable들을 찍어보면 될 듯 하다. 가설은 두 가지다.

1. 임계 구역을 선언하고 무언가가 끼어서 꼬인 다음에 임계 구역이 **해제**된다.
2. 임계 구역으로 보호되고 있어야 하는데 보호가 안 되어있다. (일찍 **해제**됐다.)

![image.png](Root%20Cause/image%206.png)

결론적으로 `ExReleaseResourceLite`를 기준으로 보면 될 것 같은데… 분기 그래프를 보면 알겠지만, `ExReleaseResourceLite` 전은 분기 수가 너무 많다. 그니까 **1**번 가설은 잠시 미루고 **2**번 가설로 가보자. 

밑쪽에 있는 `ExReleaseResourceLite` 들은 이 이후에 코드를 정리하고 종료하는 부분만 있다. 위쪽으로 가보면 이 블록이 있는데, 여기는 해제 함수 이후에 뭔가가 많다.

일단 문제가 있는 함수까지는 올바르게 찾아냈다는 걸 확인했다. 설명이 잘 와닿지가 않아서 치트키를 썼다.

<aside>

SepMakeTokenEffectiveOnly는 `UserAndGroups` 배열을 삭제+끝원소 swap 방식으로 재배치하는데, 이 작업이 락으로 보호되지 않으면 다른 스레드가 같은 배열을 동시에 읽거나 복제/해제하면서 카운트·인덱스·엔트리 포인터가 불일치하게 되고, 그 결과로 UAF / 포인터 중복으로 인해 Double Free가 발생할 수 있다.

</aside>

우선 이 함수는 `UserAndGroups` 배열에서 특정 엔트리를 제거하면서 앞으로 당긴다. Windows 토큰에는 `UserAndGroups`라는 배열(`(SID, Attributes)` 같은 구조체들의 배열)이 있고, `a1+124`는 대충 배열 길이로 생각하련다.

밑에는 이런저런 루프가 있는데 머 여기서는 그룹을 검사한다. 그리고 어딘가에서는 삭제도 한단다.

삭제로직은 이렇다고 함

<aside>

- 삭제 대상 발견
- `v3--` (전체 count 감소)
- 현재 인덱스 v2 자리에 “맨 끝(v3)의 엔트리”를 복사해서 덮어씀
- v2는 증가하지 않음(덮어쓴 새 엔트리를 다시 검사해야 하니까)
</aside>

그리고 코드 안의 특정 그룹 인덱스를 가리키는 필드들도 함께 조정한다고 한다. 대충 이동과 삭제에 따른 인덱스 보정을 해준다. 그리고 이 작업은 토큰을 수정하는 임계구역이라서 **토큰 락**으로 보호되어야 한다고 한다.

그런데 락이 없으면, 다른 스레드가 같은 토큰의 `UserAndGroups`를 읽거나, 복제하거나 등등의 작업을 동시에 진행할 수 있다. 그니까 배열이 바뀌고있고아직다안바뀌었는데, 전의 상태를 최종 상태라 생각하고 끼어드는 느낌 같다. 

- 대충 발생할 수 있는 상황들
    
    케이스 A: 인덱스/카운트 불일치 (OOB / 잘못된 엔트리 읽기)
    
    케이스 B: 엔트리 포인터의 “중복/유실” (Double Free의 씨앗)
    
    케이스 C: “이미 free된 토큰/배열을 계속 만지는” 상황
